# Experiments

Shared retrieval experimentation phase — notebooks here apply to both Project 1 (codebase RAG) and Project 2 (citizen science RAG). This folder is the foundation both projects build on before their pipelines diverge.

---

## Notebooks

| Notebook | Investigates |
|---|---|
| `01_naive_to_hybrid` | Chunking strategy, embedding model selection, hybrid retrieval construction |
| `02a_eval_bm25_mangling` | Research artifact — preserves broken whitespace BM25 tokenization to isolate the sparse leg's contribution by comparison with 02b |
| `02b_eval_baseline` | Authoritative retrieval experiments with identifier-aware BM25 tokenization — eval harness, failure mode diagnosis, LLM summary enrichment against 10 solver Q&A pairs |
| `03_qdrant_migration` | ChromaDB → Qdrant migration, point-by-point verification, hybrid parity confirmation, cross-encoder reranker preview |

---

## Contents

`qa_pairs/solvers_qa.json` — 10 Q&A pairs covering `kernelpack.solvers`. Generated by Codex. Three tiers: api, workflow, conceptual. Not yet domain-expert validated — treat recall numbers as directional.

`metadata/solvers_summaries.json` — plain-English summaries of every function in `kernelpack.solvers`, generated by Codex. Solvers module only.

`scripts/migrate_to_qdrant.py` — standalone migration script. Reads `chroma_export.pkl`, creates a Qdrant collection, and writes all points.

`utils/chunking.py` — shared chunking utilities including `bm25_tokenize`, the identifier-aware tokenizer used across 02b and 03. Import from here; do not re-implement inline.

`utils/eval_utils.py` — shared eval harness (`print_eval_table`, `is_hit`, recall computation). All notebooks use the same harness so results are directly comparable.

`reports/` — PDF reports for each experiment series.

| Report | Covers |
|---|---|
| `01_naive_to_hybrid_report.pdf` | Pipeline development: fixed-line → AST chunking → UniXcoder → hybrid RRF. VDB recommendation (ChromaDB → Qdrant). |
| `02_eval_report_v2.pdf` | Authoritative eval report: BM25 tokenization fix, v1→v3 recall attribution, wide retrieval diagnostic. |
| `03_qdrant_migration_report.pdf` | Migration verification, hybrid parity, reranker preview. |

## Authoritative Recall Numbers

All numbers from `03_qdrant_migration.ipynb`. The eval harness matches chunk IDs against `source_symbol` fields — see `utils/eval_utils.py` for the exact comparison logic before interpreting results.

| Pipeline | recall@3 | recall@5 | recall@10 |
|---|---|---|---|
| Qdrant hybrid | 6/10 | 8/10 | 8/10 |
| Qdrant hybrid + reranker | 6/10 | 6/10 | 8/10 |

**Known misses (both pipelines, both at k=10):** both are conceptual-tier queries. `build_stencil_properties` fails because "convergence order" appears in the query but not in the chunk or summary. `pu_patch_weight` fails despite vocabulary overlap — likely BM25 tokenization failure on hyphenated terms. These are not random noise; they are the concrete motivation for math-term metadata filtering.

**What the numbers mean:**
- Recall@k = fraction of queries where the correct chunk appears in the top-k results.
- The eval set covers `kernelpack.solvers` only (10 pairs). Numbers are directional until the set expands to all 6 modules and receives domain-expert validation.
- 02a is a research artifact showing broken tokenization — do not use its numbers as a baseline. 02b is the authoritative pre-Qdrant baseline.
